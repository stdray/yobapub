# Анализ миграции YobaPub на ASP.NET MVC 10 (SSR) + клиентские скрипты

## Обзор текущего стека (для контекста)

| Параметр | Значение |
|---|---|
| Архитектура | SPA, полностью клиентское приложение |
| Рендеринг | jQuery + doT.js шаблоны, вставка HTML в DOM |
| Платформа | Samsung Tizen 2.3+ (Chrome 28+), Smart TV |
| Доставка | .wgt-пакет, устанавливается на ТВ, работает офлайн (от сервера нужен только API) |
| API | REST API KinoPub (api.service-kp.com), OAuth2 device flow |
| Объём | ~2700 строк TS + 650 строк CSS, 6 страниц |

## Предлагаемый стек

| Параметр | Значение |
|---|---|
| Backend | ASP.NET Core MVC 10 (.NET 10, LTS) |
| Views | Razor (.cshtml), серверный рендеринг HTML |
| Клиентские скрипты | Vanilla JS (или минимальный TS), для фокус-навигации, плеера, клавиатуры |
| Стили | CSS (можно тот же файл) |
| Хостинг | Нужен постоянно работающий сервер (VPS/облако) |
| Доставка | TV-приложение загружает страницы с сервера по HTTP |

## Плюсы миграции на ASP.NET MVC 10 (SSR)

### 1. Серверный рендеринг -- тонкий клиент
HTML генерируется на сервере, ТВ получает готовую страницу. Это снижает нагрузку на слабый процессор Tizen 2.3 -- не нужно парсить и выполнять тяжёлый JS для построения DOM. Критично для устройств 2015-2016 годов.

### 2. Совместимость с Chrome 28 становится проще
Клиентский JS сводится к минимуму: обработка клавиш, управление фокусом, работа с `<video>`. Не нужен тяжёлый транспилятор для всего приложения -- только для небольших скриптов (или можно писать сразу на ES5).

### 3. Обновления без переустановки приложения
Сейчас для обновления UI нужно пересобрать .wgt и переустановить на каждом ТВ. С SSR-подходом UI живёт на сервере -- обновляешь серверный код, все ТВ сразу получают новую версию. Это огромное преимущество для поддержки.

### 4. Строго типизированные модели и views
Razor views с C# моделями вместо строковых doT-шаблонов. Ошибки в шаблонах ловятся на этапе компиляции, а не в runtime на ТВ, где отладка крайне затруднена.

### 5. Серверная логика для API-проксирования
API-клиент (OAuth2, refresh token, запросы к KinoPub API) переезжает на сервер. Это решает несколько проблем:
- Токены не хранятся на клиенте в localStorage (безопаснее)
- Не нужен CORS
- Можно кэшировать ответы API на сервере
- Сервер может трансформировать данные перед отправкой клиенту

### 6. Мощная экосистема .NET
Dependency injection, middleware, logging, конфигурация, health checks, кэширование -- всё из коробки. Для маленького проекта это избыточно, но упрощает рост.

### 7. Razor views хорошо подходят для "страничной" навигации
Текущая архитектура -- фактически набор страниц с mount/unmount. MVC-контроллер + Razor view -- естественное отображение этой структуры.

## Минусы и риски миграции

### 1. ФУНДАМЕНТАЛЬНАЯ ПРОБЛЕМА: Tizen-приложение -- это локальный .wgt-пакет
Tizen Web App работает из локального пакета (.wgt), который запускается в WebView. Это **не браузер с адресной строкой** -- приложение не загружает страницы с удалённого сервера по URL. Чтобы использовать SSR, нужно одно из:

**Вариант A**: Сделать приложение-обёртку (.wgt), которое открывает WebView на URL сервера. Это возможно, но:
- Требует постоянного интернет-соединения (нет офлайн-запуска)
- Начальная загрузка медленнее (HTTP round-trip вместо локального файла)
- Samsung может отклонить такое приложение в TV App Store (политика запрещает "пустые обёртки")

**Вариант B**: SSR используется только для предрендера, а готовый HTML поставляется в .wgt. Но тогда теряется главное преимущество SSR (динамический контент), и это по сути то же самое, что текущий подход.

**Вариант C**: Забыть про .wgt-пакет и запускать приложение через встроенный браузер ТВ. Но:
- На Tizen 2.3 встроенный браузер очень ограничен
- Нет доступа к Tizen API (tvinputdevice, webapis)
- Нельзя зарегистрировать медиа-клавиши пульта
- Ужасный UX: нужно вводить URL на экранной клавиатуре ТВ

### 2. Нужен постоянно работающий сервер
Сейчас приложение полностью автономно -- устанавливается на ТВ и работает. С SSR нужен сервер (VPS, Azure, AWS), который:
- Должен быть доступен 24/7
- Стоит денег (от $5-15/мес за VPS)
- Требует обслуживания (обновления, мониторинг, сертификаты)
- Создаёт единую точку отказа: сервер упал -- все ТВ перестали работать

### 3. Latency на каждую навигацию
SPA-навигация (текущий подход): переключение страницы -- мгновенное (DOM-манипуляция в памяти).
SSR-навигация: каждый переход -- HTTP-запрос к серверу + получение HTML + рендеринг. На хорошем соединении это 100-300мс, но на ТВ с Wi-Fi может быть 500мс+. Для ТВ-интерфейса, где навигация пультом должна быть мгновенной, это ощутимо.

### 4. Потеря доступа к Tizen-специфичным API
Из Tizen Web App доступны API:
- `tizen.tvinputdevice.registerKey()` -- регистрация медиа-клавиш пульта
- `webapis.productinfo` -- информация об устройстве
- `tizen.application.getCurrentApplication().exit()` -- выход из приложения

При работе через браузер (Вариант C) эти API недоступны. При обёртке (Вариант A) -- доступны, но только если JavaScript обёртки загружается локально.

### 5. Плеер всё равно остаётся на клиенте
player.ts (913 строк) -- управление `<video>`, HLS.js, субтитры, аудио-дорожки, overlay -- это полностью клиентская логика. SSR здесь не помогает. Razor view отдаст `<video>` тег, но вся работа с ним -- тот же JS, что и сейчас. Это ~35% кодовой базы, которая не упрощается.

### 6. Управление фокусом остаётся на клиенте
Навигация пультом ТВ (стрелки, Enter, Return) -- сугубо клиентская задача. Вся фокус-логика (watching.ts -- навигация по сетке карточек, serial.ts -- три зоны фокуса, settings.ts -- список + модал) переезжает в клиентские скрипты без изменений. SSR не упрощает этот код.

### 7. Смена стека команды
С TypeScript/JS/Node на C#/.NET -- это другой язык, другой инструментарий, другая ментальная модель. Если команда -- фронтендеры, переход будет болезненным.

### 8. Двойное состояние
При SSR + клиентские скрипты неизбежна проблема синхронизации состояния: сервер отрендерил HTML с одним состоянием, клиентский JS должен "подхватить" это состояние. Для ТВ-фокуса, позиции скролла, состояния плеера -- это дополнительная сложность.

### 9. OAuth2 Device Flow усложняется
Сейчас: приложение показывает код -> поллит API -> получает токен -> сохраняет в localStorage.
С SSR: нужно решить, где хранить сессию (cookies? server-side session?), как связать конкретный ТВ с сессией, как обновлять токены на сервере.

## Вердикт

Миграция на ASP.NET MVC 10 SSR **не рекомендуется** для данного проекта по следующим причинам:

1. **Архитектурный mismatch**: приложение -- Smart TV клиент, работающий из локального .wgt-пакета. SSR требует внешний сервер, что противоречит модели доставки Tizen-приложений.

2. **Половина кода не упрощается**: плеер и фокус-навигация -- чисто клиентские задачи, SSR для них бесполезен.

3. **Введение сервера создаёт проблемы**: стоимость, доступность, latency, единая точка отказа -- для приложения, которое сейчас работает полностью автономно.

4. **Трудозатраты в 2-3 раза выше**, чем миграция на Svelte, при сомнительном выигрыше.

SSR-подход имел бы смысл, если бы:
- Приложение работало в обычном браузере (не Tizen .wgt)
- Были нужны SEO, SSR-оптимизации, серверный кэш
- Планировался backend с собственной базой данных и бизнес-логикой
- Команда уже работала на C#/.NET
