# Обновление hls.js для поддержки 4K/HEVC на современных ТВ

Описание проблемы, решения, платформ и рисков. Конкретные шаги и чекбоксы —
в [hls-upgrade-tasks.md](hls-upgrade-tasks.md).

## Проблема

Текущая версия hls.js 0.14.x формально не поддерживает HEVC (H.265), однако по логам
с реального Tizen 3.0 устройства видно, что **HEVC уже работает** на текущей версии:

```
hls LEVEL_SWITCHED level=4 1920x1072 bitrate=7472613 videoCodec=hvc1.2.4.L150.B0 audioCodec=mp4a.40.2
```
(Лог: https://yobapub.3po.su/s/logs/xoqnm2gQxUOQpDF3uBYtuQ/tsv)

hls.js 0.14 не парсит HEVC-кодек-строки сам, но передаёт их «как есть» из манифеста в
MediaSource, и если браузер (Chromium 47 на Tizen 3.0) принимает `hvc1.*` через MSE —
воспроизведение работает. Фрагменты грузятся, декодируются, видео играет.

Однако есть проблемы:
- Частые `bufferStalledError` (~каждые 30 секунд) — возможно связаны с тем, что
  hls.js 0.14 некорректно оценивает буфер для HEVC-потока.
  **Подтверждено исправление на modern:** тест 2026-04-15 на том же Tizen 3.0
  с hls.js 1.6.16 (commit `5aa19e6`, лог
  https://yobapub.3po.su/s/logs/0hCxKHZVXkKjo9VAvUT8NA/tsv) — 0 stall'ов
  за всю сессию HEVC 1080p
- Отсутствие HEVC-aware ABR — hls.js 0.14 не учитывает аппаратные возможности
  при выборе уровня качества
- **4K HEVC на Tizen 3.0 не работает** — звук идёт, картинка зависает/корявая.
  Лог: https://yobapub.3po.su/s/logs/tZtBoLRIgEulyLxtvUyzEg/tsv
  Манифест: mixed HEVC+AVC, 8 levels. `pinQualityLevel` выбирает level=6
  (3840×1600 HEVC @ 2.4 Mbps) — декодер Tizen 3.0 не справляется
  с 4K через MSE. При этом 1080p HEVC работает (level=4, 1920×1072).
- В mixed-codec манифесте `pinQualityLevel` может выбрать HEVC-вариант
  вместо AVC под ту же резолюцию — нет учёта кодека при выборе

## Решение

Загружать разные версии hls.js в зависимости от платформы:
- **hls.js 0.14.x** — для legacy-устройств (Tizen 2.3, старые webOS, или по выбору пользователя)
- **hls.js 1.5+** — для современных устройств (HEVC/4K через MSE)

Это возможно благодаря текущей архитектуре: hls.js подключается как vendor-скрипт
(`externals: { 'hls.js': 'Hls' }`), весь код обращается к `window.Hls`.

## Преимущества обновления

### Технические выигрыши (общие для всех платформ)

- **Правильный парсинг HEVC-манифестов.** 0.14 пропускает `hvc1.*` «как есть» —
  работает случайно. 1.5 знает о HEVC в codec-string, корректно считает битрейт/уровень,
  учитывает это в ABR.
- **HEVC-aware ABR.** Выбор уровня качества учитывает, что HEVC при том же битрейте
  даёт лучшую картинку, но тяжелее декодируется. В mixed-codec манифесте (HEVC+AVC
  под одной резолюцией) 1.5 может предпочесть AVC при слабом CPU; 0.14 выбирает
  по голому битрейту.
- **Buffer management переписан.** Частые `bufferStalledError` каждые ~30 секунд
  на 0.14 с HEVC — именно из-за того, что оценщик буфера не понимает HEVC-фрагменты.
  В 1.5 решено: отдельные контроллеры для audio/video, корректный учёт gap'ов,
  адекватный `maxBufferLength` для high-bitrate потоков.
- **Gap jumping.** Встроенный механизм автоматического перепрыгивания дыр в буфере —
  то, что мы сейчас пытаемся делать руками через «healing seek». Потенциально
  убирает целый пласт костылей из `player.ts`; оценить точнее — после первого
  прогона modern на реальном железе.
- **Subtitle rendering.** В 1.5 нативная поддержка WebVTT через `TextTrack` API
  с корректным таймингом — возможно, упростит `player/subtitles.ts`.
- **Активная поддержка.** 0.14 — 2019 год, давно не обновляется. В 1.5+ фиксятся
  реальные баги, которые мы сейчас обходим патчами.

### Продуктовые выигрыши

- **4K HEVC на Tizen 5.5+ и современных Android TV** — главный мотиватор.
- **Меньше случайных зависаний** на 1080p HEVC для Tizen 4.0+ и Android TV
  (исчезают stalled errors).
- **Более адекватный автовыбор качества** на нестабильной сети — 1.5 быстрее
  реагирует на падение bandwidth.

## Влияние на Android TV

Android TV — отдельный случай: **WebView обновляется через Play Store независимо
от версии ОС**. Реальная версия Chromium на большинстве современных Android TV —
108–130+, не та, что дефолтно ставится с Android 9/10/11.

- **WebView ≥ 80** (почти все Android TV после 2020 года) — полноценно поддерживает
  MSE HEVC, `hvc1`/`hev1` в `isTypeSupported`, hls.js 1.5 работает штатно. Это
  **основной бенефициар** обновления — именно на Android TV 4K HEVC реально заиграет.
- **WebView 60–80** (старые коробки 2018–2019, TCL/Xiaomi бюджетки с замороженным
  системным WebView) — hls.js 1.5 соберётся и запустится, но MSE HEVC может врать
  на `isTypeSupported`. Поведение будет такое же, как сейчас на Tizen 3.0 с 4K:
  звук есть, картинка корявая. Пользователю остаётся вручную переключиться
  на «Старый телевизор».
- **WebView < 60** — 1.5 может не запуститься из-за ES2015+ фич. Автодетект должен
  выключать modern для таких случаев.

**Вывод**: Android TV — кандидат №1 на раннее включение modern. Большая установочная
база, современный WebView, минимальный риск, максимальный выигрыш.

### Важный нюанс для детекта платформы

**Для Android TV нельзя полагаться на версию Android OS.** В таблице платформ ниже
указано `Android 7–8 → WebView 51–63`, но это дефолт. Реальная версия обычно выше.
Детектить надо **именно Chrome/WebView из UA**, не Android OS:

```
navigator.userAgent  // "...; wv) ... Chrome/<version>..."
```

Если парсить версию Chrome из UA и ставить порог «Chrome ≥ 80 → modern», получается
надёжнее, чем «Android ≥ 9 → modern». Это единый механизм и для Android TV, и для
Tizen (у Tizen тоже есть Chrome-версия в UA), и для webOS — можно отказаться
от отдельных детектов per-vendor в пользу одного Chromium-порога, оставив
per-vendor только исключения (webOS 1.x–2.x на WebKit без Chrome в UA).

## Настройка «Старый телевизор»

Клиентская настройка (не серверная) в странице настроек:

- **Назначение**: принудительно использовать hls.js 0.14.x вместо автоопределения.
- **Когда нужна**: если на конкретном устройстве hls.js 1.5+ падает или работает
  нестабильно — пользователь может вручную переключиться на проверенную версию
  без потери работоспособности.
- **По умолчанию** на этапе раскатки — `true` (legacy). Автодетект Chrome-версии
  вводится отдельным PR после подтверждения modern на реальных устройствах.
- **Автоматика**: для известных legacy-платформ (Chrome < 47, webOS 1.x–2.x)
  настройка игнорируется — всегда legacy.
- **Поведение при переключении**: диалог «Настройка применится после перезапуска
  приложения. Перезапустить сейчас?» → `location.reload()`. hls.js грузится один
  раз при старте, без перезапуска переключение не имеет смысла. На Tizen
  `location.reload()` перезагружает виджет без переустановки.

## Стратегия раскатки

Ключевое ограничение: на тестовом Tizen 5.5+ **переустановка виджета невозможна**.
Modern hls.js должен попасть на устройство **штатным апдейтом виджета**, через master.

Из этого следует:

1. **Loader + адаптер + настройка должны попасть в master одновременно.** Без адаптера
   modern hls.js не запустится; без loader нечем переключать; без настройки нечем
   управлять извне.
2. **Default = legacy для всех устройств** на этапе раскатки. Никакого автодетекта
   Chrome-версии в первой итерации — только ручной opt-in через настройку
   «Старый телевизор» (по умолчанию `true`). Это исключает риск автоматической
   регрессии у живых пользователей.
3. **Автодетект — отдельным PR** после того, как modern подтверждён на Tizen 5.5+
   и Android TV.

## Риски

- **hls.js 1.5+ ES-уровень**: собирается в ES2015+. На Chromium 47+ (Tizen 3.0+) ES2015
  поддерживается нативно. На Chromium 38 (webOS 3.x) — нет, поэтому legacy.
  **Проблема для сборки**: `es-check` на `dist/release` упадёт, если modern-бандл
  пойдёт через webpack в общий `app.js`. Решение: подавать `hls-modern.min.js`
  напрямую из `node_modules/hls.js/dist/` в `vendor/` через `CopyWebpackPlugin`,
  мимо SWC, и исключить его из `escheck`.
- **Размер бандла**: hls.js 1.5 ~300KB vs 0.14 ~200KB. Некритично — грузится один раз.
- **MSE HEVC detection ненадёжен**: `MediaSource.isTypeSupported('video/mp4; codecs="hvc1.*"')`
  возвращает `true` на устройствах, которые реально не воспроизводят поток (звук есть,
  картинка — нет; см. текущую ситуацию с 4K на Tizen 3.0). Обходного пути в коде нет —
  пользователь вручную переключается на «Старый телевизор».
- ~~**Регрессия на Tizen 3.0**~~: **риск снят тестом 2026-04-15**. modern hls.js 1.6.16
  на реальном Tizen 3.0 воспроизводит HEVC 1080p стабильнее 0.14 (0 stall'ов
  за сессию против ~1/30s), `pinQualityLevel` с флагом `preferHevc` корректно
  выбирает hvc1-вариант, экономит ~40% битрейта. Tizen 3.0 становится
  кандидатом на default=modern наравне с 4.0+. Ограничение остаётся только
  на 4K HEVC — аппаратный декодер не тянет, это не про версию hls.js.
  См. запись 2026-04-15 21:46 в `doc/decision-log.md`.
- **Двойное обслуживание**: два vendor-файла, возможно два patch-скрипта. Приемлемо,
  пока не откажемся от Tizen 2.3.
- **API-диф 0.14 → 1.5+**: реальные breaking changes — переименованы события, изменён
  `Config`, структура `levels[]`, поведение `startLoad(-1)`, ABR controller переписан.
  Нужна полноценная обёртка `HlsAdapter` с двумя имплементациями — иначе `player.ts`
  и `player/panel.ts` превратятся в `if (version)`-лапшу. Это ~неделя на адаптер
  + рефакторинг `player.ts`, `player/hls.ts`, `player/panel.ts`, `utils/hls-proxy.ts`,
  `utils/hls-error.ts`.
- **Типы**: `@types/hls.js@0.13` несовместимы с типами из коробки 1.5+. Стайлгайд
  запрещает `any`, поэтому нужен свой `HlsLike`-интерфейс — пересечение того,
  что реально используется (~30–50 методов/событий).
- **`patch-hls.js`**: текущий патч завязан на структуру кода 0.14 и поломает 1.5.
  Варианты: (а) два патча, (б) убрать патч на modern и проверить, что baseline
  seek/recover не регрессирует. Риск потерять наработки из `doc/decision-log.md`
  (история «healing seek»).
- **Доступ к тест-железу**: фаза тестирования требует физические устройства
  Tizen 3.0/5.5+, webOS, Android TV. Без них фаза превращается в
  «катимся в прод и смотрим».

## Итоговая оценка

| Фаза | Работа | Оценка |
|------|--------|--------|
| 1 | Vendor-бандлы, loader, настройка «Старый телевизор» | 2–3 дня |
| 2 | `HlsAdapter` + рефакторинг player-модулей | 4–6 дней |
| 3 | HEVC/4K (серверные флаги, проверка на железе) | 2–3 дня + зависит от железа |
| 4 | Тест-матрица | упирается в доступ к устройствам |

**Итого ~2–3 недели** при наличии тестового железа. Основные точки риска:
фаза 2 (адаптер, если API разошлись сильнее ожидаемого) и доступ к Tizen 4.0+
/ 5.5+ для реальной проверки HEVC через MSE.
