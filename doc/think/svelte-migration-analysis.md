# Анализ миграции YobaPub на Svelte

## Обзор текущего стека

| Параметр | Значение |
|---|---|
| Язык | TypeScript (target ES5) |
| Рендеринг | jQuery DOM-манипуляции + doT.js шаблоны (строковая конкатенация HTML) |
| Стили | Один глобальный CSS-файл (647 строк) |
| Сборка | Webpack 5 + SWC |
| HTTP-клиент | jQuery.ajax (+ jQuery.Deferred вместо Promise) |
| Маршрутизация | Самописная (router.ts, 38 строк), стек истории в памяти |
| Состояние | Модульные переменные (`var`), без стора |
| Платформа | Samsung Tizen 2.3+ (Chrome 28+), разрешение 1920x1080 |
| Объём кода | ~2700 строк TS, ~650 строк CSS, 6 страниц |
| Страницы | login, watching, movie, serial, player, settings |

Архитектура: каждая страница -- модуль с `mount()`/`unmount()`, строит HTML через doT-шаблоны, управляет фокусом вручную через jQuery, слушает `keydown` на `window`.

## Плюсы миграции на Svelte

### 1. Декларативный UI вместо ручного DOM
Сейчас каждая страница вручную собирает HTML через строковую конкатенацию в doT-шаблонах, потом вставляет через `$root.html(...)` и управляет классами через `addClass/removeClass`. В Svelte это заменяется на декларативные `{#if}`, `{#each}`, `class:focused`, реактивные переменные -- код становится существенно читабельнее и короче.

### 2. Реактивность из коробки
Текущий паттерн: изменить переменную -> вызвать `render()` или `updateFocus()` для перерисовки. В Svelte изменение переменной автоматически обновляет DOM. Убирается вся ручная синхронизация состояния и представления.

### 3. Компонентная архитектура
Повторяющиеся элементы (Card, Button, Episode, MenuItem, SettingItem) сейчас реализованы как doT-шаблоны. В Svelte они станут полноценными компонентами с типизированными props, собственными стилями и логикой -- проще переиспользовать, тестировать и поддерживать.

### 4. Scoped CSS
Один монолитный CSS-файл на 647 строк с ручным BEM-подобным именованием. В Svelte стили можно разнести по компонентам с автоматической изоляцией -- нет риска конфликтов имён, проще удалять мёртвые стили.

### 5. Избавление от jQuery
jQuery 2.2.4 подключён как external через `<script>` и используется везде: DOM-запросы, AJAX, Deferred, события. Это ~85 КБ лишнего кода для платформы, где достаточно нативного `fetch`/`querySelector`. Svelte полностью убирает потребность в jQuery.

### 6. Меньший размер бандла
Svelte компилируется в чистый JS без runtime-библиотеки. Текущий стек тянет jQuery (~85 КБ min), doT.js, object-fit-images. После миграции итоговый бандл может стать меньше, что критично для слабых Tizen-устройств.

### 7. Упрощение управления фокусом
Сложная логика фокуса (watching.ts -- 60+ строк навигации по сетке, player.ts -- три отдельных обработчика клавиш для трёх режимов) может быть вынесена в переиспользуемые Svelte actions/stores, что упростит добавление новых страниц.

### 8. TypeScript из коробки
Svelte нативно поддерживает TypeScript в `<script lang="ts">`. Текущие типы (api.ts, app.ts) переносятся без изменений.

## Минусы и риски миграции

### 1. Совместимость с Tizen 2.3 (Chrome 28) -- ГЛАВНЫЙ РИСК
Tizen 2.3 работает на Chromium 28 (2013 год). Это означает:
- Нет `let`/`const`, стрелочных функций, `Promise`, `Map`/`Set`, template literals в нативном виде
- Svelte 5 генерирует код с использованием современного JS (Proxy, WeakMap и т.д.)
- Svelte 4 генерирует более простой код, но всё равно может использовать ES2015+ конструкции
- **Потребуется SWC/Babel для транспиляции выхода Svelte в ES5** + polyfills (core-js)
- Необходимо тщательное тестирование на реальном устройстве -- скомпилированный Svelte-код после транспиляции может вести себя иначе

### 2. HLS.js как external
HLS.js подключён через `<script>` как глобал и используется через `require('hls.js')` внутри player.ts. Нужно адаптировать импорт для Svelte-сборки (Vite/Rollup externals или dynamic import).

### 3. Прямое управление `<video>` элементом
player.ts (913 строк) интенсивно работает с HTMLVideoElement: прямые обращения к `videoEl.currentTime`, `videoEl.play()`, управление `textTracks`, создание `<track>` элементов, подписка на нативные события. Svelte не даёт особых преимуществ для этого кода -- его придётся портировать практически 1-к-1 через `bind:this` и `onMount`/`onDestroy`.

### 4. Нет готового роутера для Tizen
Текущий роутер -- 38 строк кода с простым стеком. SvelteKit избыточен и не подходит (SPA на ТВ). Придётся либо портировать текущий роутер, либо использовать лёгкую библиотеку (svelte-spa-router / tinro). Не проблема, но дополнительная работа.

### 5. Потеря инкрементальности
Миграция -- по сути переписывание. Нельзя мигрировать "по одной странице" без промежуточного слоя, потому что все страницы завязаны на общий роутер, jQuery-based API-клиент и глобальное состояние.

### 6. Время на отладку Tizen-специфики
Tizen API (`tizen.tvinputdevice`, `webapis.productinfo`) вызывается через глобальные объекты. Это продолжит работать, но любые изменения в lifecycle компонентов могут сломать регистрацию клавиш или таймеры.

### 7. Смена инструментов сборки
Svelte обычно используется с Vite (или Rollup). Текущий Webpack 5 + SWC потребует замены или добавления svelte-loader. Vite проще, но нужно настроить ES5-выход (через @vitejs/plugin-legacy или vite-plugin-svelte + SWC).

### Итого

| Метрика | Значение |
|---|---|
| Общий объём кода для переноса | ~2700 строк TS + 650 строк CSS |
| Количество Svelte-компонентов на выходе | ~15-20 (страницы + переиспользуемые: Card, Button, Episode, Sidebar, SettingItem, PlayerOverlay, PlayerMenu, Toast и т.д.) |
| Код, переносимый без изменений | ~400 строк (типы, утилиты, platform) |
| Код, требующий полного переписывания | ~1800 строк (страницы, роутер) |
| Код с заменой зависимостей | ~270 строк (API-клиент: jQuery.ajax -> fetch) |
| Оценка трудозатрат | **3-5 рабочих дней** для одного разработчика, знакомого со Svelte |
| Из них на player.ts | ~1-1.5 дня |
| Из них на отладку Tizen 2.3 | ~0.5-1 день |

### Общая сложность: **Средняя**

Проект небольшой, архитектура простая (нет сложного стейт-менеджмента, нет SSR, нет роутинга с вложенными маршрутами). Основной риск -- совместимость скомпилированного Svelte-кода с Chrome 28 на Tizen 2.3. Рекомендуется начать с proof-of-concept: собрать минимальную страницу (login) на Svelte, транспилировать в ES5 и проверить на реальном устройстве.

## Рекомендуемый план миграции (если принято решение мигрировать)

1. **PoC**: Svelte + Vite + @vitejs/plugin-legacy (или SWC), простейшая страница -> проверка на Tizen 2.3
2. **Инфраструктура**: настройка сборки, замена jQuery.ajax на fetch + polyfill, создание Svelte store для роутинга
3. **Утилиты и API**: перенос storage, platform, api-клиента (fetch-based)
4. **Страницы по порядку сложности**: login -> movie -> serial -> watching -> settings -> player
5. **Player последним**: самый сложный компонент, минимально зависит от остальных
6. **Удаление jQuery и doT**: финальная чистка после переноса всех страниц
