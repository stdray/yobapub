# Анализ миграции бэкенда YobaPub.Proxy на Bun + TypeScript

## Обзор текущего стека

| Параметр | Значение |
|---|---|
| Фреймворк | ASP.NET Core (.NET 10 preview) |
| Язык | C# |
| Архитектура | Reverse proxy + HLS rewriter + admin-панель |
| База данных | LiteDB (embedded NoSQL) |
| Аутентификация | Cookie auth (ASP.NET Identity) |
| Объём | ~500 строк логики, 10 файлов |
| Тесты | HlsRewriterTests (xUnit) |

## Предлагаемый стек

| Параметр | Значение |
|---|---|
| Рантайм | Bun |
| Язык | TypeScript |
| HTTP-сервер | `Bun.serve()` |
| База данных | SQLite (`bun:sqlite`, встроен в Bun) |
| Аутентификация | Cookie auth (ручная подпись) |
| Тесты | `bun:test` |

---

## Плюсы миграции

### 1. Единый язык для фронта и бэка
Фронтенд уже на TypeScript. Единый стек упрощает жизнь одному разработчику: один язык, один набор инструментов, возможность шарить типы между фронтом и бэком.

### 2. Избавление от .NET SDK
.NET 10 -- preview, SDK тяжёлый (~500 МБ). Bun -- один бинарник (~50 МБ), быстрая установка, нативная поддержка TypeScript без дополнительной настройки.

### 3. Мгновенный старт и лёгкий деплой
Bun стартует мгновенно, нет JIT-прогрева. Docker-образ на порядок меньше (`oven/bun:alpine` ~100 МБ vs `mcr.microsoft.com/dotnet/aspnet` ~200+ МБ). Можно деплоить и без Docker -- просто `bun run server.ts`.

### 4. Меньше boilerplate
ASP.NET middleware pipeline, DI-контейнер, конфигурация через `IOptions<T>` -- всё это overkill для маленького прокси. В Bun простой роутер + fetch -- десятки строк вместо сотен.

### 5. Нативный fetch для проксирования
`fetch()` встроен в Bun, стриминг ответов через `Response` -- проксирование пишется в пару строк. Не нужен `HttpClient` с ручной настройкой.

### 6. HLS rewriter переносится 1:1
Regex-логика переписывания m3u8 манифестов -- чистая функция без зависимостей. JS regex не уступает .NET для этой задачи. Тестовые фикстуры (m3u8-файлы) переносятся без изменений.

### 7. SQLite вместо LiteDB
`bun:sqlite` встроен в рантайм, не нужна внешняя зависимость. SQLite более зрелый, с лучшим тулингом и документацией.

## Минусы миграции

### 1. Замена LiteDB на SQLite
LiteDB -- document-oriented, SQLite -- реляционная. Нужно спроектировать таблицу, написать SQL-запросы, реализовать CRUD и retention. Модель данных простая (логи), поэтому миграция несложная, но требует работы.

### 2. Cookie auth руками
ASP.NET даёт cookie authentication из коробки: подпись, шифрование, expiry, CSRF-токены. В Bun придётся реализовать самостоятельно: подпись cookie (HMAC), проверку, CSRF-защиту. Для 2 защищённых эндпоинтов это посильно, но нужно сделать правильно.

### 3. Зрелость Bun для streaming/proxy
Bun стабилен для HTTP, но edge-cases в проксировании (backpressure при стриминге больших ответов, обработка chunked encoding, таймауты) могут потребовать отладки. ASP.NET здесь battle-tested.

### 4. Меньше гарантий типизации в middleware
ASP.NET pipeline строго типизирован: middleware, DI, конфигурация проверяются на этапе компиляции. В Bun/TS роутинг и middleware менее формализованы, больше полагаются на runtime.

### 5. Нет фонового сервиса из коробки
`LogRetentionService` в .NET -- `BackgroundService` с `PeriodicTimer`. В Bun -- `setInterval()`, что функционально достаточно, но нет graceful shutdown из коробки (нужно слушать `process.on('SIGTERM')`).

### 6. Переписывание тестов
HlsRewriterTests нужно перенести на `bun:test`. Фикстуры (m3u8-файлы) переносятся as-is, но тестовый код переписывается.

## Оценка сложности по компонентам

| Компонент | Сложность | Оценка времени | Комментарий |
|---|---|---|---|
| Reverse proxy (`/v1/*`, `/oauth2/*`, `/proxy`) | Низкая | 2-3 часа | `fetch()` + проброс заголовков, whitelist хостов |
| HLS rewriter (`/hls/rewrite`) | Низкая | 1-2 часа | Regex 1:1, чистая функция |
| Static files + SPA fallback | Тривиально | 30 мин | `Bun.serve()` static option |
| Log storage (LiteDB -> SQLite) | Средняя | 3-4 часа | Схема таблицы, CRUD, пагинация, фильтрация |
| Log retention (BackgroundService) | Низкая | 30 мин | `setInterval()` + SQL DELETE |
| Cookie auth + admin endpoints | Средняя | 3-4 часа | HMAC-подпись, login/logout, CSRF |
| Admin HTML-страницы | Низкая | 1-2 часа | Template literals (2 страницы) |
| Конфигурация | Низкая | 30 мин | JSON + env переменные |
| Тесты HlsRewriter | Низкая | 1-2 часа | `bun:test`, фикстуры те же |
| Эндпоинты (`/api/log`, `/api/proxy-config`, assetlinks) | Низкая | 1 час | Простые JSON-обработчики |

### Итого: 2-3 дня работы

## Вопрос шаблонизации админки

Сейчас admin-панель рендерится на сервере через Razor. Варианты для Bun:

1. **Template literals в коде** -- админка маленькая (логин + просмотр логов), HTML собирается строками. Никаких зависимостей, аналогично фронтовым doT-шаблонам. Самый простой вариант.
2. **Статичные HTML + fetch** -- HTML-файлы с JS, данные загружаются через API. Админка становится мини-SPA.
3. **Лёгкий шаблонизатор** (eta, mustache, doT) -- overkill для 2 страниц.

Рекомендация: **template literals**. Минимум кода, не нужны зависимости, легко поддерживать.

## Рекомендация

Миграция оправдана если:
- Хочется единого TypeScript-стека для одного разработчика
- .NET SDK (preview) -- лишняя зависимость в проекте
- Планируется шарить типы/код между фронтом и бэком

Миграция не оправдана если:
- Бэкенд работает стабильно и не планируется его развивать
- Важна battle-tested надёжность проксирования и стриминга
