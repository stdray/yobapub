# План: обновление hls.js для поддержки 4K/HEVC на современных ТВ

## Проблема

Текущая версия hls.js 0.14.x формально не поддерживает HEVC (H.265), однако по логам
с реального Tizen 3.0 устройства видно, что **HEVC уже работает** на текущей версии:

```
hls LEVEL_SWITCHED level=4 1920x1072 bitrate=7472613 videoCodec=hvc1.2.4.L150.B0 audioCodec=mp4a.40.2
```
(Лог: https://yobapub.3po.su/s/logs/xoqnm2gQxUOQpDF3uBYtuQ/tsv)

hls.js 0.14 не парсит HEVC-кодек-строки сам, но передаёт их «как есть» из манифеста в
MediaSource, и если браузер (Chromium 47 на Tizen 3.0) принимает `hvc1.*` через MSE —
воспроизведение работает. Фрагменты грузятся, декодируются, видео играет.

Однако есть проблемы:
- Частые `bufferStalledError` (~каждые 30 секунд) — возможно связаны с тем, что
  hls.js 0.14 некорректно оценивает буфер для HEVC-потока
- Отсутствие HEVC-aware ABR — hls.js 0.14 не учитывает аппаратные возможности
  при выборе уровня качества
- **4K HEVC на Tizen 3.0 не работает** — звук идёт, картинка зависает/корявая.
  Лог: https://yobapub.3po.su/s/logs/tZtBoLRIgEulyLxtvUyzEg/tsv
  Манифест: mixed HEVC+AVC, 8 levels. `pinQualityLevel` выбирает level=6
  (3840×1600 HEVC @ 2.4 Mbps) — декодер Tizen 3.0 не справляется
  с 4K через MSE. При этом 1080p HEVC работает (level=4, 1920×1072).
- В mixed-codec манифесте `pinQualityLevel` может выбрать HEVC-вариант
  вместо AVC под ту же резолюцию — нет учёта кодека при выборе

## Решение

Загружать разные версии hls.js в зависимости от платформы:
- **hls.js 0.14.x** — для legacy-устройств (Tizen 2.3, старые webOS, или по выбору пользователя)
- **hls.js 1.5+** — для современных устройств (HEVC/4K через MSE)

Это возможно благодаря текущей архитектуре: hls.js подключается как vendor-скрипт
(`externals: { 'hls.js': 'Hls' }`), весь код обращается к `window.Hls`.

## Преимущества обновления

### Технические выигрыши (общие для всех платформ)

- **Правильный парсинг HEVC-манифестов.** 0.14 пропускает `hvc1.*` «как есть» —
  работает случайно. 1.5 знает о HEVC в codec-string, корректно считает битрейт/уровень,
  учитывает это в ABR.
- **HEVC-aware ABR.** Выбор уровня качества учитывает, что HEVC при том же битрейте
  даёт лучшую картинку, но тяжелее декодируется. В mixed-codec манифесте (HEVC+AVC
  под одной резолюцией) 1.5 может предпочесть AVC при слабом CPU; 0.14 выбирает
  по голому битрейту.
- **Buffer management переписан.** Частые `bufferStalledError` каждые ~30 секунд
  на 0.14 с HEVC — именно из-за того, что оценщик буфера не понимает HEVC-фрагменты.
  В 1.5 решено: отдельные контроллеры для audio/video, корректный учёт gap'ов,
  адекватный `maxBufferLength` для high-bitrate потоков.
- **Gap jumping.** Встроенный механизм автоматического перепрыгивания дыр в буфере —
  то, что мы сейчас пытаемся делать руками через «healing seek». Потенциально
  убирает целый пласт костылей из `player.ts`; оценить точнее — после фазы 0.
- **Subtitle rendering.** В 1.5 нативная поддержка WebVTT через `TextTrack` API
  с корректным таймингом — возможно, упростит `player/subtitles.ts`.
- **Активная поддержка.** 0.14 — 2019 год, давно не обновляется. В 1.5+ фиксятся
  реальные баги, которые мы сейчас обходим патчами.

### Продуктовые выигрыши

- **4K HEVC на Tizen 5.5+ и современных Android TV** — главный мотиватор.
- **Меньше случайных зависаний** на 1080p HEVC для Tizen 4.0+ и Android TV
  (исчезают stalled errors).
- **Более адекватный автовыбор качества** на нестабильной сети — 1.5 быстрее
  реагирует на падение bandwidth.

## Влияние на Android TV

Android TV — отдельный случай: **WebView обновляется через Play Store независимо
от версии ОС**. Реальная версия Chromium на большинстве современных Android TV —
108–130+, не та, что дефолтно ставится с Android 9/10/11.

- **WebView ≥ 80** (почти все Android TV после 2020 года) — полноценно поддерживает
  MSE HEVC, `hvc1`/`hev1` в `isTypeSupported`, hls.js 1.5 работает штатно. Это
  **основной бенефициар** обновления — именно на Android TV 4K HEVC реально заиграет.
- **WebView 60–80** (старые коробки 2018–2019, TCL/Xiaomi бюджетки с замороженным
  системным WebView) — hls.js 1.5 соберётся и запустится, но MSE HEVC может врать
  на `isTypeSupported`. Поведение будет такое же, как сейчас на Tizen 3.0 с 4K:
  звук есть, картинка корявая. Нужен тот же fallback-механизм, что описан в фазе 3.
- **WebView < 60** — 1.5 может не запуститься из-за ES2015+ фич. Автодетект должен
  выключать modern для таких случаев.

**Вывод**: Android TV — кандидат №1 на раннее включение modern. Большая установочная
база, современный WebView, минимальный риск, максимальный выигрыш.

### Важный нюанс для детекта платформы

**Для Android TV нельзя полагаться на версию Android OS.** В таблице платформ ниже
указано `Android 7–8 → WebView 51–63`, но это дефолт. Реальная версия обычно выше.
Детектить надо **именно Chrome/WebView из UA**, не Android OS:

```
navigator.userAgent  // "...; wv) ... Chrome/<version>..."
```

Если парсить версию Chrome из UA и ставить порог «Chrome ≥ 80 → modern», получается
надёжнее, чем «Android ≥ 9 → modern». Это единый механизм и для Android TV, и для
Tizen (у Tizen тоже есть Chrome-версия в UA), и для webOS — можно отказаться
от отдельных детектов per-vendor в пользу одного Chromium-порога, оставив
per-vendor только исключения (webOS 1.x–2.x на WebKit без Chrome в UA).

## Настройка «Старый телевизор»

Добавить в страницу настроек пункт **«Старый телевизор»** (клиентская настройка,
не серверная):

- **Назначение**: принудительно использовать hls.js 0.14.x вместо автоопределения.
- **Когда нужна**: если на конкретном устройстве hls.js 1.5+ падает или работает
  нестабильно — пользователь может вручную переключиться на проверенную версию
  без потери работоспособности.
- **Реализация**: `storage.ts` → `isLegacyHls(): boolean`, по умолчанию `false`.
  Когда `true` — загружается `hls-legacy.min.js` и отключаются HEVC-фичи
  (эквивалент текущего поведения).
- **Автоматика**: для известных legacy-платформ (Tizen <3, webOS <4) настройка
  игнорируется — всегда legacy.

## План работ

### Фаза 1 — подготовка (без изменения поведения)

1. **Исследовать API-совместимость hls.js 0.14 vs 1.5+**
   - Составить список используемых API: `Events.*`, `Config`, `levels`, `currentLevel`,
     `recoverMediaError()`, `startLoad()`, `stopLoad()`, `destroy()`
   - Найти breaking changes между версиями
   - Определить, нужен ли абстрактный интерфейс-обёртка или API достаточно совместимы

2. **Подготовить vendor-бандлы**
   - `vendor/hls-legacy.min.js` — текущий 0.14.x (с диагностическими патчами)
   - `vendor/hls-modern.min.js` — hls.js 1.5+ (минифицированный)
   - Обновить `CopyWebpackPlugin` в `webpack.config.js`

3. **Добавить настройку «Старый телевизор»**
   - `storage.ts`: `isLegacyHls()` / `setLegacyHls()`
   - `settings.ts`: новый пункт в клиентских настройках (не серверных)

### Фаза 2 — динамическая загрузка

4. **Реализовать выбор версии hls.js**
   - Убрать `<script src="vendor/hls.min.js">` из `index.html`
   - Добавить динамический загрузчик: определяет платформу →
     выбирает `hls-legacy` или `hls-modern` → загружает `<script>` →
     `onload` запускает приложение
   - Логика выбора (по Chrome-версии из UA, единый механизм для всех вендоров):
     - `isLegacyHls() === true` → legacy
     - Chrome < 47 (нет MSE HEVC, ES2015 может не работать) → legacy
     - Tizen 3.0 (Chrome 47) — legacy по умолчанию: HEVC 1080p уже работает
       на 0.14, переключение на 1.5 ради нестабильного 4K — регрессия
     - Chrome ≥ 56 (Tizen ≥ 4.0, webOS ≥ 4.x, Android TV WebView ≥ 56) → modern
     - webOS 1.x–2.x (WebKit, нет Chrome в UA) → legacy как исключение

5. **Адаптировать `patch-hls.js`**
   - Текущий патч работает только с 0.14. Для 1.5+ либо сделать отдельный патч,
     либо убрать (в новых версиях seekToStartPos уже управляется лучше)

6. **Обновить типы**
   - Текущие `@types/hls.js@0.13.3` не покрывают 1.5+. hls.js 1.5+ поставляется
     с собственными типами. Решение: использовать пересечение типов или minimal
     interface для общего API

### Фаза 3 — HEVC/4K

7. **Включить серверные флаги на современных ТВ**
   - При загрузке настроек (`/v1/device/settings`) проверить, что `supportHevc`,
     `supportHdr`, `support4k` включены (или подсказать пользователю включить)

8. **Проверить hls4 стриминг с HEVC**
   - На Tizen 3.0+ с hls.js 1.5+ выбрать 4K-контент
   - Убедиться, что hls.js парсит HEVC-манифест, pinQualityLevel работает,
     видео декодируется

9. **Обработка fallback**
   - Если hls.js 1.5+ не может воспроизвести HEVC-стрим (устройство рапортует MSE
     HEVC, но реально не тянет) — показать понятную ошибку с рекомендацией
     снизить качество или включить «Старый телевизор»

### Фаза 4 — тестирование

См. раздел **«План ручного тестирования»** ниже.

## Стратегия раскатки

Ключевое ограничение: на тестовом Tizen 5.5+ **переустановка виджета невозможна**.
Значит шим-эксперимент с подменой `vendor/hls.min.js` на устройстве не работает —
modern hls.js должен попасть на устройство **штатным апдейтом виджета**, через master.

Из этого следует:

1. **Loader + адаптер + настройка должны попасть в master одновременно.** Без адаптера
   modern hls.js не запустится; без loader нечем переключать; без настройки нечем
   управлять извне.
2. **Default = legacy для всех устройств** на этапе раскатки. Никакого автодетекта
   Chrome-версии в первой итерации — только ручной opt-in через настройку
   «Старый телевизор» (по умолчанию `true`). Это исключает риск автоматической
   регрессии у живых пользователей.
3. **Автодетект — отдельным PR** после того, как modern подтверждён на Tizen 5.5+
   и Android TV.

### Поведение настройки «Старый телевизор»

- Хранится в `localStorage` через `storage.ts` (`isLegacyHls()` / `setLegacyHls()`),
  default = `true`.
- При переключении в `settings.ts`: показать диалог **«Настройка применится после
  перезапуска приложения. Перезапустить сейчас?»** → по подтверждению вызвать
  `location.reload()`. На Tizen `location.reload()` перезагружает виджет без
  переустановки — именно то, что нужно.
- hls.js грузится один раз при старте через `<script>`-тег до `main.ts`,
  в рантайме `window.Hls` не подменяется. Поэтому без перезапуска переключение
  не имеет смысла.

### Диагностические логи (обязательно перед раскаткой)

Чтобы лог тестировщика был самодостаточным, добавить три лога:

1. **Стартовый probe окружения** (один раз при `main.ts`):
   ```
   [env] ua=<navigator.userAgent>
   [env] tizen=<tizen.systeminfo.platform.version или 'n/a'>
   [env] mse hvc1=<isTypeSupported('video/mp4; codecs="hvc1.1.6.L150.B0"')>
   [env] mse hev1=<isTypeSupported('video/mp4; codecs="hev1.1.6.L150.B0"')>
   [env] mse avc1=<isTypeSupported('video/mp4; codecs="avc1.640028"')>
   ```
2. **Лог инициализации плеера**: `[hls] version=<Hls.version> mode=<legacy|modern> isSupported=<bool>`.
3. **Лог переключения настройки**: `[settings] legacy hls toggled: <old>→<new>, reloading`.

Эти три лога превращают «тестировщик что-то нажимал» в восстанавливаемую
из дампа историю сеанса.

## План ручного тестирования

### Этап 0 — Baseline до раскатки (своё железо)

**Цель**: зафиксировать текущее поведение Tizen 2.3, Tizen 3.0 и Android TV
на текущем `master`, чтобы после раскатки сравнить.

Делать **до** мержа hls-upgrade в master.

Сценарий на каждом устройстве:
1. Логин → Watching → деталь сериала → плеер.
2. 1080p H.264, 5 минут без взаимодействия, потом 5 seek'ов
   (+30с, −30с, +2 мин, в конец, в начало). Закрыть.
3. **Только Tizen 3.0 и Android TV**: 1080p HEVC, 10 минут без взаимодействия.
   Посчитать частоту `bufferStalledError` в логах.
4. Записать URL логов в `doc/decision-log.md` с пометкой
   `baseline pre-hls-upgrade, <device>, <date>`.

**Время**: ~40 минут на все три устройства.

### Этап 1 — Раскатка в master с default=legacy

После того как loader, адаптер, настройка и три диагностических лога готовы:

1. Собрать release с `isLegacyHls() === true` по умолчанию.
2. **Tizen 2.3**: повторить полный сценарий Этапа 0. Любая регрессия — блокер,
   modern даже не успел включиться, всё должно быть идентично.
3. **Tizen 3.0**: то же самое (default legacy → 0.14, должно быть как раньше).
4. **Android TV**: то же самое (default legacy → 0.14).
5. Если всё PASS — мержим в master, релизим виджет на Tizen 5.5+ через штатный
   апдейт.

**Что смотреть**: ничего не должно поменяться по сравнению с baseline. Если
поменялось — loader или setting сломал legacy путь, modern даже не трогали.

**Время**: 30 минут.

### Этап 2 — Параллельная проверка modern на своём железе

Пока тестировщик не доступен, на своём Tizen 3.0 и Android TV вручную
выключить «Старый телевизор» → перезапуск → пройти **сценарии A, B, C, E**
из инструкции тестировщика (см. ниже). 4K (сценарий D) на Tizen 3.0 пропускаем —

**Главный вопрос на Tizen 3.0**: запустился ли вообще hls.js 1.5 на Chrome 47.
Если да — сравнить частоту stalled errors с baseline Этапа 0. Гипотеза:
в HEVC 1080p ошибок станет сильно меньше.

**На Android TV**: то же + попытка 4K в сценарии D. Если приставка тянет 4K
через MSE — это первое подтверждение основной фичи ещё до Tizen 5.5.

**Время**: ~1 час.

> **Сценарий C — Full HD HEVC сериал (10 минут).**
> Открыть любой сериал 2023–2024 года, любую серию, качество **1080p**. Play.
> **Смотреть 10 минут подряд, не трогая пульт.**
> *Что должен увидеть*: видео идёт без зависаний и чёрных кадров. Если зависает —
> записать **точное время** (минута:секунда от начала воспроизведения), это
> поможет найти место в логе.
>
> **Сценарий D — 4K HEVC (10 минут) — главный сценарий.**
> Открыть свежий фильм с 4K (помечен **2160p** в выборе качества). Выбрать
> **2160p**. Play. **Смотреть 5 минут.** Потом одна перемотка вперёд на 5 минут,
> ещё 3 минуты воспроизведения.
> *Что должен увидеть*: картинка показывается в полном разрешении (не пиксельная
> мазня), нет «звук есть, видео фриз», нет чёрных кадров. Если есть проблема —
> точное время.
>
> **Сценарий E — переключение серий (5 минут).**
> Открыть сериал → серию → перемотать к концу (за 30 сек до конца) → дождаться
> автоперехода → в следующей серии посмотреть 1 минуту → закрыть.
> *Что должен увидеть*: автопереход срабатывает, marktime сохраняется
> (повторное открытие — с того же места).
>
> **Сценарий F — возврат на legacy.**
> Settings → **«Старый телевизор» включить** → перезапуск → повторить **только
> сценарий D** (4K). На legacy 4K не работает — это ожидаемо, картинка должна
> быть кривая или плеер должен ругнуться. Это нужно, чтобы убедиться, что
> переключение настройки реально сменило версию hls.js.
>
> Сообщить: «всё ОК» / «проблема в сценарии X на минуте Y».

---

**Что смотреть в логах после прохода тестировщика**:

| Сценарий | На что смотреть | PASS-критерий |
|----------|----------------|---------------|
| Стартовый лог | `[hls] version=1.5.x mode=modern` после выключения настройки | версия сменилась |
| A | ошибки навигации, фокуса | нет ошибок в логе |
| B | `bufferStalledError` на 1080p H.264 | нет или единичные |
| C | частота `bufferStalledError` на 1080p HEVC | **≈0**, было ~1 за 30 сек на 0.14 |
| D | `bufferStalledError`, `mediaError`, `fragLoadError`, отказ декодера на 4K | нет ошибок, видео идёт |
| E | `MEDIA_ATTACHED`, `MANIFEST_PARSED`, marktime | события приходят, marktime в API |
| F | `[hls] version=0.14.x mode=legacy` после включения настройки | версия откатилась |

Если C показывает PASS, а D — FAIL, значит modern починил buffer management,
но MSE 4K не тянет даже Tizen 5.5. Это не блокер для мержа — выигрыш на 1080p
HEVC уже окупает обновление. 4K — вторая итерация с fallback на качество ниже.

**Время тестировщика**: 40–50 минут одним сеансом.
**Время разбора логов**: 30–60 минут.

### Этап 4 — Решение об автодетекте

После Этапов 1–3 в `doc/decision-log.md` должна сложиться картина:

- Tizen 5.5+ + modern: PASS / FAIL по 4K, PASS / FAIL по 1080p HEVC
- Tizen 3.0 + modern: лучше / хуже / так же, чем legacy
- Android TV + modern: PASS / FAIL по 4K
- Tizen 2.3 + legacy: без регрессий

На основе этих данных — отдельным PR — включить автодетект Chrome-версии и
сменить default. Опция «Старый телевизор» остаётся как escape hatch для тех,
у кого автодетект ошибся.

### Что записывать каждый раз

Для каждого сеанса в `doc/decision-log.md`:
- Дата, устройство (модель + версия OS + версия Chrome из стартового лога),
  git SHA сборки.
- Этап, сценарий, версия hls.js (legacy/modern из стартового лога плеера).
- URL логов на `yobapub.3po.su/s/logs`.
- PASS / FAIL / PARTIAL + что именно. Для HEVC — частота stalled errors в минуту.

## Риски

- **hls.js 1.5+ ES-уровень**: собирается в ES2015+. На Chromium 47+ (Tizen 3.0+) ES2015
  поддерживается нативно. На Chromium 38 (webOS 3.x) — нет, поэтому legacy.
  **Проблема для сборки**: `es-check` на `dist/release` упадёт, если modern-бандл
  пойдёт через webpack в общий `app.js`. Решение: подавать `hls-modern.min.js`
  напрямую из `node_modules/hls.js/dist/` в `vendor/` через `CopyWebpackPlugin`,
  мимо SWC, и исключить его из `escheck`.
- **Размер бандла**: hls.js 1.5 ~300KB vs 0.14 ~200KB. Некритично — грузится один раз.
- **MSE HEVC detection ненадёжен**: `MediaSource.isTypeSupported('video/mp4; codecs="hvc1.*"')`
  возвращает `true` на устройствах, которые реально не воспроизводят поток (звук есть,
  картинка — нет; см. текущую ситуацию с 4K на Tizen 3.0). Fallback (фаза 3.9)
  критичен и должен быть не «понятная ошибка», а автоматическое понижение уровня
  качества + запоминание инвалидного уровня per-device.
- **Регрессия на Tizen 3.0**: сейчас HEVC 1080p **работает** на 0.14 (подтверждено
  логами). Переключение Tizen 3.0 на 1.5+ может сломать то, что уже работает,
  ради 4K, который всё равно не тянет декодер. Порог автовыбора должен быть
  **Tizen ≥ 4.0**, не ≥ 3.0. Для 3.0 — остаётся legacy по умолчанию, modern только
  как опциональный тумблер.
- **Двойное обслуживание**: два vendor-файла, возможно два patch-скрипта. Приемлемо,
  пока не откажемся от Tizen 2.3.
- **Доступ к тест-железу**: фаза 4 требует физические устройства Tizen 3.0/5.5+,
  webOS, Android TV. Без них фаза превращается в «катимся в прод и смотрим».

## Оценка сложности и трудозатрат

### Что реально просто (1–2 дня каждое)

- Динамическая загрузка vendor-скрипта через `<script>` + `onload`. Архитектура уже
  это позволяет: `externals: { 'hls.js': 'Hls' }`, весь код обращается к `window.Hls`,
  статических `import`-ов hls.js внутри кода нет.
- Настройка «Старый телевизор» в `storage.ts` + пункт в `settings.ts` — шаблонная работа.
- Детект Tizen версии через `tizen.systeminfo` / UA.

### Что сложнее, чем выглядит в плане

1. **API-диф 0.14 → 1.5+ (пункт 1 фазы 1 занижен).** Между этими версиями реальные
   breaking changes: переименованы события, изменён `Config`, структура `levels[]`,
   поведение `startLoad(-1)`, ABR controller переписан. Нужна не «таблица API»,
   а полноценная обёртка `HlsAdapter` с двумя имплементациями — иначе `player.ts`
   и `player/panel.ts` превратятся в `if (version)`-лапшу. Это ~неделя на адаптер
   + рефакторинг `player.ts`, `player/hls.ts`, `player/panel.ts`,
   `utils/hls-proxy.ts`, `utils/hls-error.ts`.

2. **Типы.** `@types/hls.js@0.13` несовместимы с типами из коробки 1.5+. Стайлгайд
   запрещает `any`, поэтому нужен свой `HlsLike`-интерфейс — пересечение того,
   что реально используется (~30–50 методов/событий). Делается руками по мере
   рефакторинга.

3. **`patch-hls.js`.** Текущий патч завязан на структуру кода 0.14 и поломает 1.5.
   Варианты: (а) два патча, (б) убрать патч на modern и проверить, что baseline
   seek/recover не регрессирует. Риск потерять наработки из `doc/decision-log.md`
   (история «healing seek»).

### Рекомендуемая фаза 0 (эксперимент до рефакторинга)

Перед тем как вводить адаптер, проектировать интерфейсы и трогать `player.ts`:

- Собрать `hls-modern.min.js` как vendor-файл (скопировать из `node_modules/hls.js/dist/`).
- На одном устройстве Tizen ≥ 4.0 вручную подменить `<script src="vendor/hls.min.js">`
  на modern-версию и запустить существующий код без изменений.
- Зафиксировать реальный список несовместимостей (что упало, что работает).

Это отвечает на главный вопрос «насколько API совместимы» за день, **до** любого
проектирования адаптера. Адаптер-интерфейс вводить только после того, как шим
покажет реальный список расхождений — иначе он спроектируется по догадкам.

### Итоговая оценка

| Фаза | Работа | Оценка |
|------|--------|--------|
| 0 | Шим-эксперимент на реальном железе | 1 день |
| 1 | Vendor-бандлы, loader, настройка «Старый телевизор» | 2–3 дня |
| 2 | `HlsAdapter` + рефакторинг player-модулей | 4–6 дней |
| 3 | HEVC/4K, fallback на инвалидный уровень качества | 2–3 дня + зависит от железа |
| 4 | Тест-матрица | упирается в доступ к устройствам |

**Итого ~2–3 недели** при наличии тестового железа. Основные точки риска:
фаза 2 (адаптер, если API разошлись сильнее ожидаемого) и доступ к Tizen 4.0+
/ 5.5+ для реальной проверки HEVC через MSE.
