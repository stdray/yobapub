# Проблемы воспроизведения на Tizen 2.3 (Chromium 28)

## Баги Chromium 28 + MSE

Samsung Tizen 2.3 использует Chromium 28 с особенностями реализации Media Source Extensions.

### 1. `buffered` пустой при старте, но работает после загрузки фрагментов
При старте воспроизведения `buffered=[none]` даже при `readyState=4`. После загрузки фрагментов `buffered` заполняется корректно (например `76.8-81.6,691.2-753.6,1243.2-1252.8`).
**Подтверждено логами 2026-04-01:** `video playing ct=0 readyState=4 buffered=[none]` при старте, затем `video seeked ct=1244.13 readyState=4 buffered=76.8-81.6,691.2-753.6,1243.2-1252.8` после загрузки.

### 2. `readyState` врёт при старте
Сообщает `HAVE_ENOUGH_DATA` (4) при `buffered=[none]` — данных нет, но readyState говорит что достаточно. После загрузки фрагментов readyState корректен.
**Подтверждено логами 2026-04-01:** `readyState=4 buffered=[none]` на старте.

### 3. `play()` на зависшем видео — no-op (не подтверждено)
Если видео не на паузе (`paused === false`), но зависло (stalled), повторный вызов `play()` ничего не делает. Требует проверки.

### 4. Seek при старте не работает (hls.js не подхватывает)
При старте `videoEl.currentTime = pos` выполняется, но hls.js продолжает грузить с sn=1 (start=0). Seek на играющем видео после загрузки нескольких фрагментов работает корректно — hls.js подхватывает новую позицию.
**Подтверждено логами 2026-04-01:** `onSourceReady pos=1634` → doSeek ставит currentTime=1634 → hls.js грузит sn=1,2,3... с начала. Но ручной seek к 700 и 1244 после нескольких фрагментов работает: hls.js грузит sn=70,71... рядом с целевой позицией.

## Проблема: возобновление воспроизведения с сохранённой позиции

При открытии видео с `state.position > 0` воспроизведение должно начаться с сохранённой позиции. Вместо этого начинается с 0 или зависает.

### Предпосылки

На Tizen 2.3 (Chromium 28) есть основания считать, что прямой старт воспроизведения с произвольной позиции может не работать корректно через MSE. Поэтому рассматривается стратегия: начать воспроизведение с позиции 0 (гарантированно работает), а затем сделать seek к нужной позиции.

### Почему seek после старта тоже сложен

hls.js 0.14.x игнорирует `startPosition` в конфиге — всегда загружает с sn=1 (start=0). Подтверждено логами FRAG_LOADING: при `startPosition=489` первый загруженный фрагмент — sn=1 start=0.

Значит, после старта с 0 нужен явный seek к сохранённой позиции. Но данных на этой позиции ещё нет в момент старта, и от момента seek зависит, сработает он или нет.

### Подход 1: doSeek на playing/canplay (текущий, коммит HEAD)

```
onSourceReady → play() → ждём playing/canplay → videoEl.currentTime = pos
```

**Результат:** seek молча фейлится (баг #4). hls.js не реагирует на seek — продолжает грузить с начала (sn=1, sn=2, sn=3...). Видео играет с 0.

**Почему не работает:** В момент doSeek hls.js ещё не полностью инициализирован (загружен только манифест, ни один фрагмент не буферизован). Stream controller не реагирует на изменение currentTime.

**Плюсы:** Простой код. Ручная перемотка и пауза/плей работают нормально.
**Минусы:** Возобновление с позиции не работает — всегда стартует с 0.

### Подход 2: trySeekThenPlay — seek, потом play

```
onSourceReady → videoEl.currentTime = pos → ждём seeked → play()
```

**Результат:** seeked не приходит. Видео зависает на нулевой позиции навсегда.

**Почему не работает:** Данных на позиции `pos` нет (загружен только sn=1 start=0). Seek на позицию без данных тихо фейлится (баг #4), seeked не приходит, ожидание вечное.

### Подход 3: FRAG_BUFFERED → play() → seek

```
onSourceReady → play() → ждём FRAG_BUFFERED рядом с pos → play() → setTimeout 200ms → if diff > 2: videoEl.currentTime = pos
```

**Результат:** Частично работает. hls.js загружает фрагменты по порядку (sn=1, sn=2...) — FRAG_BUFFERED рядом с pos=489 пришёл бы через долгое время (50+ фрагментов). С фильтром `fragEnd >= pos - 5` ожидание затягивается.

**Проблема с diff > 2:** После play() на sn=1 currentTime=0, setTimeout 200ms делает seek currentTime=489, но данных на 489 нет → тихий фейл (баг #4). Без `diff > 2` гарда — повторный seek на ту же позицию вызывает вечный seeking=true.

**Проблема с позицией:** После safePlay currentTime автоматически переходит к первому загруженному фрагменту. Если фрагмент рядом с pos буферизован, currentTime оказывается рядом с pos и seek не нужен. Но FRAG_BUFFERED фильтр ждёт конкретный фрагмент — если он загружен не в порядке рядом с pos, ожидание затягивается.

### Подход 4 (не пробовали): FRAG_BUFFERED первый → seek на playing видео

```
onSourceReady → play() → ждём ЛЮБОЙ первый FRAG_BUFFERED → videoEl.currentTime = pos
```

**Гипотеза:** После первого буферизованного фрагмента hls.js полностью инициализирован. Seek на играющем видео вызовет hls.js перезагрузить фрагменты вокруг новой позиции (как при ручной перемотке).

**Обоснование:** Ручная перемотка работает — `continuePlaying` просто ставит `videoEl.currentTime = pos`, и hls.js подхватывает новую позицию. Разница в том, что к моменту ручной перемотки hls.js уже загрузил несколько фрагментов и стабильно работает.

## Проблема: ручная перемотка — позиция скачет назад

После ручного seek (стрелками) позиция currentTime кратковременно показывает новую позицию, потом скачет к старой.

### Что происходит

1. `applySeek` → `continuePlaying({position: 574})` → `videoEl.currentTime = 574`
2. hls.js начинает грузить фрагменты около 574 (sn=58, sn=59...)
3. Пока фрагменты грузятся, видео показывает старый кадр
4. `video playing` приходит с currentTime=574 (баг #4 — геттер врёт)
5. При нажатии пауза/плей currentTime скачет к реальной позиции (~42-48), где были последние декодированные данные
6. hls.js ещё не успел буферизовать данные на 574

### Почему это проблема

`continuePlaying` при `needSeek` просто ставит `videoEl.currentTime = pos` без ожидания загрузки данных. Для ручной перемотки это обычно работает, потому что hls.js грузит фрагменты быстро. Но на медленном соединении или при большом скачке позиции — видно скачок.

В текущей версии (HEAD) ручная перемотка работает стабильно — скачки редки.

## Проблема: bufferNudgeOnStall дрифт позиции

hls.js при обнаружении stall сдвигает currentTime вперёд на маленькие интервалы (nudge). На Chromium 28 stall-детекция часто ложно срабатывает (из-за пустого buffered). В результате позиция постепенно уходит вперёд.

### Решение

`nudgeMaxRetry = 0` в конфиге hls.js полностью отключает nudge. Протестировано — стабильнее без nudge.

**Текущий статус:** в HEAD стоит `nudgeMaxRetry = 10` (дефолт). В одной из итераций ставили 0 — помогало.

## Что работает стабильно (текущий HEAD)

1. **Запуск воспроизведения с начала** — play() на canplay/MANIFEST_PARSED, без seek
2. **Ручная перемотка** — `videoEl.currentTime = pos` на играющем видео, hls.js подхватывает
3. **Пауза/плей** — `videoEl.pause()` / `videoEl.play()`
4. **Смена качества/аудио** — пересоздание hls.js с новым URL
5. **FRAG_LOADING логирование** — диагностика загрузки фрагментов
