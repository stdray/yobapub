# Проблемы воспроизведения на Tizen 2.3 (Chromium 28)

## Tizen 3.0 vs Tizen 2.3

На **Tizen 3.0** (Chromium 38, AppleWebKit/538.1) resume с позиции работает без костылей:
- `startPosition` в hls.js работает — фрагменты грузятся рядом с целевой позицией (sn=126 start=1249 при pos=1257)
- `buffered` корректен с самого старта
- `readyState` адекватен
- doSeek из onSourceReady корректно доводит до точной позиции
- Ручная перемотка, пауза/плей — всё работает

**Подтверждено логами 2026-04-01 (Tizen 3.0):** `playSource startPosition=1257` → hls.js грузит sn=126 start=1249 → `video seeked ct=1252.76` → `video playing ct=1252.76`. Второй старт: `startPosition=1739` → sn=174 start=1729 → работает.

Все проблемы ниже — **только Tizen 2.3** (Chromium 28).

---

## Особенности Chromium 28 + MSE

### 1. `buffered` пустой при старте, потом работает
При старте воспроизведения `buffered=[none]` даже при `readyState=4`. После загрузки фрагментов `buffered` заполняется корректно (например `76.8-81.6,691.2-753.6,1243.2-1252.8`).
**Подтверждено логами 2026-04-01 (Tizen 2.3):** `video playing ct=0 readyState=4 buffered=[none]` при старте, затем `video seeked ct=1244.13 readyState=4 buffered=76.8-81.6,691.2-753.6,1243.2-1252.8` после загрузки.

### 2. `readyState` врёт при старте
Сообщает `HAVE_ENOUGH_DATA` (4) при `buffered=[none]` — данных нет, но readyState говорит что достаточно. После загрузки фрагментов readyState корректен.

### 3. `play()` на зависшем видео — no-op (не подтверждено)
Если видео не на паузе (`paused === false`), но зависло (stalled), повторный вызов `play()` ничего не делает. Требует проверки.

### 4. hls.js `startPosition` игнорируется
hls.js 0.14.x на Chromium 28 игнорирует `startPosition` — всегда загружает с sn=1 (start=0). На Tizen 3.0 `startPosition` работает корректно.
**Подтверждено логами 2026-04-01 (Tizen 2.3):** `playSource startPosition=1634` → hls.js грузит sn=1 start=0, sn=2 start=14.4... с самого начала.

### 5. Seek при раннем вызове не подхватывается hls.js
`videoEl.currentTime = pos` сразу после старта (на playing/canplay) — hls.js не реагирует и продолжает грузить с начала. Seek на играющем видео после загрузки нескольких фрагментов работает корректно — hls.js подхватывает новую позицию.
**Подтверждено логами 2026-04-01 (Tizen 2.3):** doSeek ставит currentTime=1634 → hls.js грузит sn=1,2,3... Но ручной seek к 700 и 1244 после нескольких фрагментов работает: hls.js грузит sn=70,71... рядом с целевой позицией.

## Проблема: возобновление воспроизведения с сохранённой позиции

При открытии видео с `state.position > 0` воспроизведение должно начаться с сохранённой позиции. На Tizen 2.3 вместо этого начинается с 0.

### Предпосылки

На Tizen 3.0 resume работает из коробки — `startPosition` в hls.js корректно стартует загрузку с нужного фрагмента.

На Tizen 2.3 `startPosition` игнорируется (баг #4). Стратегия: начать воспроизведение с позиции 0 (гарантированно работает), дождаться готовности hls.js, затем seek к нужной позиции. Ручной seek после загрузки фрагментов работает стабильно — нужно воспроизвести те же условия при старте.

### Подход 1: doSeek на playing/canplay (текущий, коммит HEAD)

```
onSourceReady → play() → ждём playing/canplay → videoEl.currentTime = pos
```

**Результат:** seek не срабатывает (баг #5). hls.js не реагирует на seek — продолжает грузить с начала (sn=1, sn=2, sn=3...). Видео играет с 0.

**Почему не работает:** В момент doSeek hls.js ещё не полностью инициализирован (загружен только манифест, ни один фрагмент не буферизован). Stream controller не реагирует на изменение currentTime.

**Плюсы:** Простой код. Ручная перемотка и пауза/плей работают нормально. На Tizen 3.0 resume тоже работает.
**Минусы:** На Tizen 2.3 возобновление с позиции не работает — всегда стартует с 0.

### Подход 2: trySeekThenPlay — seek, потом play

```
onSourceReady → videoEl.currentTime = pos → ждём seeked → play()
```

**Результат:** seeked не приходит. Видео зависает на нулевой позиции навсегда.

**Почему не работает:** hls.js только что стартовал, фрагментов нет. Seek не подхватывается (баг #5), seeked не приходит, ожидание вечное.

### Подход 3: FRAG_BUFFERED рядом с pos → play() → seek

```
onSourceReady → play() → ждём FRAG_BUFFERED рядом с pos → play() → setTimeout 200ms → if diff > 2: videoEl.currentTime = pos
```

**Результат:** Не работает. hls.js загружает фрагменты по порядку с начала (sn=1, sn=2...) — FRAG_BUFFERED рядом с pos=489 пришёл бы через долгое время (50+ фрагментов).

### Подход 4 (не пробовали): первый FRAG_BUFFERED → seek

```
onSourceReady → play() → ждём ЛЮБОЙ первый FRAG_BUFFERED → videoEl.currentTime = pos
```

**Гипотеза:** После первого буферизованного фрагмента hls.js полностью инициализирован. Seek на играющем видео вызовет hls.js перезагрузить фрагменты вокруг новой позиции (как при ручной перемотке).

**Обоснование:** Ручная перемотка работает — `continuePlaying` просто ставит `videoEl.currentTime = pos`, и hls.js подхватывает новую позицию. Разница в том, что к моменту ручной перемотки hls.js уже загрузил несколько фрагментов и стабильно работает.

## Проблема: bufferNudgeOnStall дрифт позиции

hls.js при обнаружении stall сдвигает currentTime вперёд на маленькие интервалы (nudge). На Chromium 28 stall-детекция часто ложно срабатывает (из-за пустого buffered при старте). В результате позиция может уходить вперёд.

### Решение

`nudgeMaxRetry = 0` в конфиге hls.js полностью отключает nudge. Протестировано — стабильнее без nudge.

**Текущий статус:** в HEAD стоит `nudgeMaxRetry = 10` (дефолт). В одной из итераций ставили 0 — помогало.

## Что работает стабильно (текущий HEAD)

### На обеих платформах
1. **Запуск воспроизведения с начала** — play() на MANIFEST_PARSED
2. **Ручная перемотка** — `videoEl.currentTime = pos` на играющем видео, hls.js подхватывает
3. **Пауза/плей** — `videoEl.pause()` / `videoEl.play()`
4. **Смена качества/аудио** — пересоздание hls.js с новым URL

### Только Tizen 3.0
5. **Resume с позиции** — `startPosition` работает, doSeek доводит до точной позиции

### Не работает на Tizen 2.3
6. **Resume с позиции** — `startPosition` игнорируется, ранний seek не подхватывается hls.js
